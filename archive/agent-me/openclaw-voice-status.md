# OpenClaw 语音对话现状与规划

> **核心问题：为什么 OpenClaw 目前还没有原生支持语音对话？**
>
> > 日期：2026-03-29

---

## 一、当前 OpenClaw 的能力边界

### 1.1 现有的交互方式

```
OpenClaw 当前支持的交互：

1. Telegram Bot (文本)
   - 用户发送文字消息
   - OpenClaw 回复文字
   - 支持 Markdown、按钮等

2. Web 界面 (文本)
   - 浏览器中的聊天界面
   - 基于文本的输入输出

3. CLI 命令行 (文本)
   - 终端交互
   - 适合开发者

4. API 调用 (程序)
   - 其他系统调用 OpenClaw
   - 程序化交互

缺失：
❌ 原生语音输入
❌ 原生语音输出
❌ 实时语音流
```

### 1.2 为什么没有原生语音？

```yaml
架构设计原因:

历史包袱:
    - OpenClaw 最初设计为"文本优先"
    - 基于消息队列架构
    - 适合异步文本处理

技术复杂度:
    - 语音需要实时流处理
    - 需要音频采集和播放
    - 需要 STT/TTS 引擎集成
    - 延迟要求极高 (<1s)

资源限制:
    - 语音模型较大 (VibeVoice 0.5B)
    - 需要 GPU/专用芯片支持
    - 移动端和桌面端差异大

设计哲学:
    - OpenClaw 专注于"Agent 大脑"
    - 不处理"感官输入"
    - 语音属于"感官层"
```

## 二、语音对话的技术挑战

### 2.1 实时性要求

```
文本对话 vs 语音对话：

文本对话：
用户打字 ──▶ 发送 ──▶ 处理 ──▶ 回复
  │          │        │        │
  │          │        │        └── 显示文字
  │          │        └── LLM 推理 (几秒)
  │          └── 网络传输
  └── 用户思考时间 (不确定)

语音对话：
用户说话 ──▶ 识别 ──▶ 处理 ──▶ 合成 ──▶ 播放
  │          │        │        │        │
  │          │        │        │        └── 即时播放
  │          │        │        └── TTS (几百ms)
  │          │        └── LLM 推理 (必须 <1s)
  │          └── STT (几百ms)
  └── 实时采集

挑战：
- 每段延迟都必须 < 几百毫秒
- 总延迟必须 < 1-2 秒
- 需要流式处理
```

### 2.2 技术栈复杂度

```
完整的语音对话系统需要：

┌─────────────────────────────────────────┐
│           语音采集层                      │
│  - 麦克风访问权限                          │
│  - 音频预处理 (降噪、回声消除)              │
│  - 音频编码 (Opus/PCM)                    │
└─────────────────────────────────────────┘
           │
           ▼
┌─────────────────────────────────────────┐
│           语音识别层 (STT)                │
│  - VibeVoice ASR 或 Whisper             │
│  - 实时流式识别                          │
│  - 说话人识别 (可选)                      │
└─────────────────────────────────────────┘
           │
           ▼
┌─────────────────────────────────────────┐
│           Agent 大脑层 (OpenClaw)         │
│  - 意图理解                              │
│  - 上下文管理                            │
│  - LLM 推理                              │
│  - 工具调用                              │
└─────────────────────────────────────────┘
           │
           ▼
┌─────────────────────────────────────────┐
│           语音合成层 (TTS)                │
│  - VibeVoice Realtime 或 Edge TTS       │
│  - 流式语音生成                          │
│  - 声音克隆 (可选)                        │
└─────────────────────────────────────────┘
           │
           ▼
┌─────────────────────────────────────────┐
│           语音播放层                      │
│  - 音频解码                              │
│  - 播放器管理                            │
│  - 打断检测 (可选)                        │
└─────────────────────────────────────────┘

复杂度：5 个独立层，每层的延迟都要控制在几百毫秒内
```

## 三、解决方案：分层架构

### 3.1 为什么不把语音做进 OpenClaw？

```
设计决策：分离关注点

OpenClaw 核心 (保持不变):
├── Agent 大脑
├── 工具调用
├── 记忆管理
├── 链上交互
└── 文本接口

Agent Me (新增语音层):
├── 语音采集
├── STT (VibeVoice)
├── 连接 OpenClaw
├── TTS (VibeVoice)
└── 语音播放

好处：
1. 各司其职：OpenClaw 专注智能，Agent Me 专注交互
2. 可替换：可以换不同的语音引擎
3. 平台适配：移动端、桌面端可以不同实现
4. 独立演进：语音技术快速发展，可以独立更新
```

### 3.2 Agent Me 作为语音前端

```
用户 ──▶ Agent Me App
           │
           ├── 语音采集 (手机麦克风)
           │
           ├── STT (VibeVoice ASR)
           │   └── "帮我查一下今天的日程"
           │
           ├── WebSocket 发送
           ▼
OpenClaw ──▶ 处理请求
           │   ├── 理解意图
           │   ├── 查询日历
           │   └── 生成回复
           │
           ├── WebSocket 返回
           ▼
Agent Me ──▶ "你今天有 3 个会议..."
           │
           ├── TTS (VibeVoice Realtime)
           │
           └── 语音播放 (手机扬声器)

关键：OpenClaw 只处理文本，Agent Me 处理语音转换
```

## 四、当前可行的临时方案

### 4.1 方案 1：手机语音输入法

```
最简单的 workaround：

用户操作：
1. 打开 Telegram
2. 点击语音输入按钮
3. 说话 → 自动转文字
4. 发送给 OpenClaw
5. 阅读 OpenClaw 的回复

优点：
- 立即可以使用
- 不需要开发

缺点：
- 体验不连贯
- 没有语音回复
- 需要手动操作
```

### 4.2 方案 2：浏览器语音 API

```
技术方案：
- 浏览器 Web Speech API
- 语音转文字 (Chrome 内置)
- 文字转语音 (浏览器 TTS)

实现：
- 开发一个 Web 界面
- 集成 Web Speech API
- 连接 OpenClaw WebSocket

优点：
- 跨平台 (任何有浏览器的设备)
- 无需安装 App

缺点：
- 质量一般 (依赖浏览器)
- 不支持打断
- 延迟较高
```

### 4.3 方案 3：现有语音助手桥接

```
使用 Siri / Google Assistant 作为桥梁：

配置：
- "Hey Siri, 问我的 Agent..."
- Siri 接收语音
- 通过 Shortcuts 调用 OpenClaw API
- 返回结果给 Siri 朗读

优点：
- 使用系统级语音
- 支持后台唤醒

缺点：
- 配置复杂
- 受限于 Siri 能力
- 延迟高
```

## 五、长期解决方案：Agent Me

### 5.1 开发计划

```yaml
Phase 1: MVP (2-4 周)
  目标: 证明可行性

  技术栈:
    - React Native / Flutter
    - WebSocket 连接 OpenClaw
    - 系统 STT/TTS (临时)

  功能:
    - 基础语音对话
    - 文字显示
    - 基础历史记录

Phase 2: 生产版本 (2-3 个月)
  目标: 高质量体验

  技术栈:
    - VibeVoice ASR (本地/云端)
    - VibeVoice Realtime (本地)
    - 后台保活
    - 省电优化

  功能:
    - 高质量语音
    - 低延迟 (<1s)
    - 后台常驻
    - 通知推送

Phase 3: 完善体验 (3-6 个月)
  目标: 超越现有方案

  功能:
    - 语音唤醒 "Hey Agent"
    - 打断支持
    - 多设备同步
    - 离线支持
```

### 5.2 技术选型

```
语音引擎选择:

方案 A: VibeVoice (推荐)
  ASR: VibeVoice-ASR-7B (云端) 或 Whisper (本地)
  TTS: VibeVoice-Realtime-0.5B (本地)
  优点: 质量好，开源，支持长对话
  缺点: 需要集成，模型较大

方案 B: 云端 API (快速)
  ASR: Google Speech-to-Text / Azure Speech
  TTS: Google TTS / Azure TTS
  优点: 快速集成，质量稳定
  缺点: 有成本，隐私问题，依赖网络

方案 C: 混合模式 (平衡)
  在线时: VibeVoice 云端
  离线时: Whisper 本地 + Edge TTS
  优点: 平衡质量和成本
  缺点: 复杂度较高

推荐: 方案 C，从云端开始，逐步迁移到本地
```

## 六、FAQ

### Q1: 为什么不让 OpenClaw 直接支持语音？

```
A: 架构设计选择

OpenClaw 的核心价值是"Agent 大脑"：
- 工具调用
- 记忆管理
- 链上交互
- 逻辑推理

语音是"交互层"，应该由专门的前端处理：
- 不同设备需要不同实现
- 语音技术快速迭代
- 保持核心简单

类比：
- OpenClaw = 电脑主机
- Agent Me = 显示器 + 键盘 + 鼠标
- 你可以换不同的显示器，主机不变
```

### Q2: 现在有什么临时方案可以用？

```
A: 三个方案

1. 手机语音输入法 + Telegram
   - 立即可用
   - 体验一般

2. 浏览器 Web Speech API
   - 需要开发简单界面
   - 跨平台

3. 等待 Agent Me MVP
   - 预计 2-4 周
   - 最佳体验
```

### Q3: Agent Me 会开源吗？

```
A: 商业模式考虑

OpenClaw (开源):
- 核心 Agent 运行时
- 保持开源中立
- 社区驱动

Agent Me (商业):
- 提供连接服务
- 订阅制收费
- 专业支持

但可以考虑：
- 开源基础 SDK
- 商业版提供增值服务
```

## 七、总结

### 7.1 现状

```
OpenClaw 当前: 文本交互 ✅
OpenClaw 语音: 原生不支持 ❌
原因: 架构设计分离关注点
```

### 7.2 解决方案

```
短期: 语音输入法 + Telegram (临时)
中期: Agent Me MVP (2-4 周)
长期: Agent Me 生产版 (2-3 个月)
```

### 7.3 关键洞察

> **OpenClaw 专注于做最好的"Agent 大脑"，语音交互交给专门的前端。这样每个部分都可以做到最好，用户也能获得最佳体验。**
