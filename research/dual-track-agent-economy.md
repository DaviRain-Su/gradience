# 双轨 Agent 经济：质押竞争 vs 能力竞争

> **核心设计：DeFi 任务用质押，普通任务用能力，ERC-8004 记录统一声誉**
> 
> 讨论日期：2026-03-29

---

## 一、双轨设计架构

### 1.1 两种模式的定位

```
┌─────────────────────────────────────────────────┐
│              Agent Layer 双轨经济                │
├─────────────────────────────────────────────────┤
│                                                 │
│  轨道 A: 质押竞争（Staking Competition）         │
│  ├── 适用场景：DeFi、金融、高价值任务            │
│  ├── 核心逻辑：质押越多，责任越大，收益越高      │
│  ├── 风险控制：质押罚没机制                      │
│  └── 例子：资产管理、借贷清算、大额交易          │
│                                                 │
├─────────────────────────────────────────────────┤
│                                                 │
│  轨道 B: 能力竞争（Capability Competition）      │
│  ├── 适用场景：普通任务、创意、代码              │
│  ├── 核心逻辑：结果说话，质量决定收益            │
│  ├── 风险控制：声誉系统（ERC-8004）              │
│  └── 例子：写代码、设计、数据分析、内容创作      │
│                                                 │
├─────────────────────────────────────────────────┤
│                                                 │
│  统一层：ERC-8004 身份与声誉                     │
│  ├── 两种模式共享同一身份体系                    │
│  ├── 声誉跨模式累积                              │
│  └── 诚实度是核心指标                            │
│                                                 │
└─────────────────────────────────────────────────┘
```

### 1.2 为什么需要双轨？

```yaml
质押竞争的优势:
  - 适合高价值任务
  - 经济约束直接有效
  - 风险可量化（质押金额 = 最大损失）
  
质押竞争的劣势:
  - 门槛高（需要资金）
  - 不适合新手
  - 资本可能垄断

能力竞争的优势:
  - 门槛低（任何人可参与）
  - 鼓励真实能力
  - 更公平（不看出身看结果）

能力竞争的劣势:
  - 验证成本高（需要评判）
  - 早期声誉积累慢
  - 单次任务风险

双轨并存:
  ✅ 覆盖不同场景
  ✅ 给用户选择
  ✅ 风险分层管理
```

## 二、能力竞争 + ERC-8004 诚实度系统

### 2.1 核心机制

```
诚实度（Honesty Score）设计：

初始值：每个新 Agent 诚实度 = 50（满分 100）

诚实度提升：
├── 任务成功完成：+ 5 分
├── 获得好评：+ 3 分
├── 连续成功（无失败）：+ 2 分/次
└── 长期活跃：+ 1 分/月

诚实度降低：
├── 任务失败（未完成）：- 10 分
├── 质量不达标：- 5 分
├── 被举报并确认：- 20 分
├── 恶意行为：- 50 分（并可能封号）
└── 长时间不活跃：- 2 分/月

诚实度用途：
├── 70+ 分：可以接高价值任务
├── 50-70 分：普通任务
├── 30-50 分：只能接低价值任务
├── < 30 分：限制参与，需要"冷却期"
└── < 10 分：永久封禁
```

### 2.2 ERC-8004 如何记录诚实度

```solidity
// ERC-8004 扩展：诚实度记录

interface IAgentReputation {
    struct Reputation {
        uint256 honestyScore;      // 诚实度 0-100
        uint256 totalTasks;        // 总任务数
        uint256 successfulTasks;   // 成功任务数
        uint256 failedTasks;       // 失败任务数
        uint256 lastUpdateTime;    // 最后更新时间
        bytes32[] attestationCIDs; // 历史证明 CID 列表
    }
    
    // 更新诚实度（由任务合约调用）
    function updateHonesty(
        address agent,
        bool success,
        uint256 qualityScore,  // 质量评分 0-100
        bytes32 attestationCID // 本次任务的证明 CID
    ) external;
    
    // 查询诚实度
    function getHonesty(address agent) external view returns (uint256);
    
    // 查询完整声誉
    function getReputation(address agent) external view returns (Reputation memory);
}

// 实现合约
contract AgentReputation is IAgentReputation {
    mapping(address => Reputation) public reputations;
    
    function updateHonesty(
        address agent,
        bool success,
        uint256 qualityScore,
        bytes32 attestationCID
    ) external {
        // 只有授权的任务合约可以调用
        require(authorizedContracts[msg.sender], "Not authorized");
        
        Reputation storage rep = reputations[agent];
        
        if (success) {
            // 成功：根据质量加分
            uint256 bonus = qualityScore / 20; // 质量 100 分 = +5 分
            rep.honestyScore = min(rep.honestyScore + 5 + bonus, 100);
            rep.successfulTasks++;
        } else {
            // 失败：扣分
            rep.honestyScore = rep.honestyScore > 10 ? rep.honestyScore - 10 : 0;
            rep.failedTasks++;
        }
        
        rep.totalTasks++;
        rep.lastUpdateTime = block.timestamp;
        rep.attestationCIDs.push(attestationCID);
        
        emit HonestyUpdated(agent, rep.honestyScore, success);
    }
}
```

### 2.3 诚实度的抗操纵设计

```yaml
防止刷分:
  问题: Agent 自己发布简单任务，自己完成，刷高诚实度
  
  解决:
    - 任务发布者也有诚实度要求
    - 低诚实度发布者发布的任务不计入声誉
    - 需要第三方真实用户确认

防止早期诚实后期作恶:
  问题: Agent 前期老实积累声誉，后期一次性骗大钱
  
  解决:
    - 时间衰减：旧任务权重递减
    - 近期表现权重更高
    - 大额任务需要额外验证

防止评判员偏袒:
  问题: 评判员给某些 Agent 高分
  
  解决:
    - 评判员也有诚实度
    - 多评判员随机分配
    - 评判结果公开可审计

防止冷冻账号复活:
  问题: 被封账号重新注册
  
  解决:
    - DID 绑定设备/身份
    - 新账号有观察期
    - 行为模式分析
```

## 三、两种模式的切换与结合

### 3.1 任务如何选择模式？

```solidity
// 任务配置

struct TaskConfig {
    uint256 reward;
    uint256 minHonesty;      // 最低诚实度要求
    uint256 requiredStake;   // 需要质押的金额（0 表示不需要）
    TaskMode mode;           // 模式选择
}

enum TaskMode {
    CAPABILITY_ONLY,     // 只用能力竞争（无质押）
    STAKE_ONLY,          // 只用质押竞争
    HYBRID               // 混合：需要质押 + 诚实度要求
}

// 任务发布者选择
function createTask(TaskConfig memory config) external {
    if (config.reward < 100) {
        // 小额任务：只用能力竞争
        config.mode = TaskMode.CAPABILITY_ONLY;
        config.requiredStake = 0;
        config.minHonesty = 30;  // 较低门槛
    } else if (config.reward > 10000) {
        // 大额任务：需要质押 + 高诚实度
        config.mode = TaskMode.HYBRID;
        config.requiredStake = config.reward / 10;  // 10% 质押
        config.minHonesty = 80;  // 高要求
    } else {
        // 中等任务：默认能力竞争
        config.mode = TaskMode.CAPABILITY_ONLY;
        config.minHonesty = 50;
    }
}
```

### 3.2 Agent 如何在两种模式间转换？

```
新手路径（能力竞争 → 质押竞争）：

阶段 1：新手期
├── 诚实度：50（初始）
├── 只能接小额任务（能力竞争）
├── 积累成功记录
└── 目标：诚实度提升到 70+

阶段 2：成长期
├── 诚实度：70+
├── 可以接中等任务
├── 开始有小额质押任务资格
└── 目标：建立声誉 + 积累资金

阶段 3：成熟期
├── 诚实度：90+
├── 有大量资金
├── 可以接高价值质押任务
└── 成为平台核心 Agent

优势：
- 低门槛进入（不需要资金）
- 凭能力上升
- 两条路径可以互通
```

### 3.3 声誉的跨模式价值

```yaml
能力竞争的声誉在质押模式的价值:
  - 高诚实度 Agent 可以减少质押比例
  - 例子：诚实度 90+ 的 Agent，质押要求减半
  - 激励：做好能力任务，降低质押门槛

质押模式的记录在能力模式的价值:
  - 成功完成质押任务，诚实度也提升
  - 大额任务成功，诚实度大幅提升
  - 激励：做好质押任务，提升能力任务资格

统一身份的好处:
  - 一个 Agent 可以在两种模式下同时活动
  - 声誉共享，互相强化
  - 用户看到一个综合评分
```

## 四、完整经济循环

### 4.1 能力竞争循环

```
用户发布任务（小额，无质押）
    │
    ▼
多个 Agent 竞争（凭能力）
    │
    ▼
评判员评分（质量决定）
    │
    ▼
最佳 Agent 获胜，获得奖励
    │
    ▼
ERC-8004 更新诚实度（成功 +5 分）
    │
    ▼
诚实度提升 → 可以接更高价值任务
    │
    └──── 循环 ────▶

核心：诚实工作 → 声誉提升 → 更多机会
```

### 4.2 质押竞争循环

```
用户发布任务（大额，需质押）
    │
    ▼
Agent 质押参与（资金门槛）
    │
    ▼
完成或失败（结果决定）
    │
    ├── 成功：获得奖励 + 返还质押 + 诚实度 +10
    │
    └── 失败：质押罚没 + 诚实度 -20
    │
    ▼
ERC-8004 记录结果
    │
    ▼
长期诚实：高声誉 + 资金累积
    │
    └──── 循环 ────▶

核心：质押约束 + 成功奖励 = 风险与收益平衡
```

### 4.3 两个循环的交互

```
Agent A：
├── 先做能力任务（小额）
├── 诚实度提升到 80
├── 获得质押任务资格（质押要求降低）
├── 完成大额质押任务
├── 诚实度提升到 95
└── 成为平台顶级 Agent

Agent B：
├── 有资金，直接做质押任务
├── 成功，诚实度提升
├── 同时做能力任务（ reputation 更高，容易中标）
└── 两条线并进

协同效应：
- 能力任务为质押任务筛选人才
- 质押任务为能力任务提供上升通道
- 统一声誉系统连接两个世界
```

## 五、可能的问题与解答

### 5.1 诚实度能代替质押吗？

```
不能完全代替：

小额任务：
  - 诚实度足够（损失有限）
  - 验证成本可接受
  
大额任务：
  - 需要经济约束（质押）
  - 诚实度只能降低概率，不能消除风险
  - 质押提供即时补偿（罚没）

最佳组合：
- 小额：纯诚实度
- 中额：诚实度 + 小额质押
- 大额：高诚实度 + 大额质押
```

### 5.2 新手如何开始？

```
新手友好设计：

1. 初始诚实度 50（不是 0）
   - 给新手机会
   - 可以接简单任务

2. 新手专区
   - 专门给新手的任务
   - 低要求，低奖励
   - 快速积累前 10 个成功记录

3. 导师制度
   - 高声誉 Agent 可以带新人
   - 新人成功，导师也获得奖励
   - 加速新手成长

4. 观察期保护
   - 前 5 个任务失败不扣分（只记录）
   - 降低新手压力
```

### 5.3 如何应对"声誉攻击"？

```
攻击方式：恶意给竞争对手打低分

防御：

1. 评判员随机分配
   - 无法预知谁会评判
   - 难以预谋攻击

2. 评判员诚实度
   - 恶意评判降低评判员诚实度
   - 失去评判资格

3. 多评判员共识
   - 单一评判员不能决定结果
   - 需要多数同意

4. 上诉机制
   - Agent 可以申诉
   - 更高层评判（成本更高，更公正）
```

## 六、总结

### 6.1 核心设计

```
双轨 Agent 经济：

质押竞争（DeFi 任务）          能力竞争（普通任务）
     │                              │
     │    统一 ERC-8004 身份         │
     └────────────┬─────────────────┘
                  │
           诚实度系统
           - 成功 +5 分
           - 失败 -10 分
           - 记录不可篡改
                  │
     ┌────────────┴─────────────────┐
     │                              │
高诚实度降低质押要求            高诚实度解锁高价值任务
     │                              │
     └────────────┬─────────────────┘
                  │
           目标：诚实工作最赚钱
```

### 6.2 关键优势

```yaml
低门槛:
  - 新手可以从能力竞争开始
  - 不需要资金就可以参与
  - 凭能力上升

高天花板:
  - 有能力 + 有资金可以做大额任务
  - 质押竞争收益更高
  - 两条路径互通

安全:
  - 小额任务靠声誉
  - 大额任务靠质押
  - 风险分层管理

公平:
  - 不看出身看结果
  - 诚实工作累积声誉
  - 长期主义获利
```

### 6.3 一句话总结

> **质押竞争和能力竞争是 Agent 经济的"双轨"：DeFi 任务用资金约束风险，普通任务用声誉筛选质量，ERC-8004 统一记录诚实度，让两条轨道互通，最终目标一致——让诚实工作成为最优策略。**
