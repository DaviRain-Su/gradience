# GRA-252: Gateway drive() 分治重构 + 持久化事件游标

## 目标

将 `apps/agent-daemon/src/gateway/gateway.ts` 中 ~170 行的 `drive()` 大泥球拆分为按状态分治的 strategy handlers，并把 `PollingMarketplaceEventListener` 的内存去重游标持久化到 SQLite，解决 daemon 重启后事件丢失/重复的问题。

## 现状

- `drive()` 是一个 `while(true)` 嵌套所有状态转换的 imperative 脚本
- 任意一步失败（如 `apply` 超时）整个 purchase 直接 FAILED，没有分级恢复
- `PollingMarketplaceEventListener` 使用内存 `Set<string>` 做 `processedSignatures`，daemon 重启即丢失
- 架构评审报告已标记为 Finding 4 + Finding 6

## 实现要求

### 1. Per-State Handler 架构

- 定义统一接口 `StateHandler { handle(record): Promise<StateTransitionResult> }`
- 实现 `PendingHandler`, `TaskCreatingHandler`, `TaskCreatedHandler`, `ApplyingHandler`, `ExecutingHandler`, `SettlingHandler`
- `gateway.ts` 中的 `drive()` 变成 router：根据 `record.status` 调用对应 handler，处理返回的 `nextState` + `delayMs`

### 2. 持久化 Event Cursor

- 在 SQLite `gateway_purchases` 同库中新建表 `gateway_meta(key TEXT PRIMARY KEY, value TEXT)`
- 存储键：`last_block_time` (INTEGER), `last_signature` (TEXT)
- `PollingMarketplaceEventListener.start()` 时读取 cursor；`onEvent` 成功处理后更新 cursor
- daemon 重启后从 cursor 恢复，避免跳过有效事件

### 3. 分级重试

- `FAILED` 状态不再只能整体重试；部分 handler（如 `TaskCreatingHandler`）内部支持带 jitter 的 backoff 重试
- `maxRetries` 仍作为全局安全上限

## 验收标准

- [ ] `drive()` 行数 < 50，变成纯 router
- [ ] 每个 handler 可独立单元测试
- [ ] SQLite 中存在 `gateway_meta` 表且包含 cursor 记录
- [ ] daemon 重启后 event listener 不重复处理已消费事件
- [ ] `gateway.test.ts` 通过；新增 `handler.test.ts` 覆盖各状态 handler
- [ ] `npm run build` 在 `apps/agent-daemon` 通过
