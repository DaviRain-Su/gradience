# GRA-202: [Arena] Support multi-judge quorum or programmatic judge pool

## Description

Extend Agent Arena's judge model from a **single designated judge** to a **programmatic judge pool** that supports multi-judge quorums. This increases fairness for high-stakes tasks and reduces reliance on any single judge's availability or bias.

Two modes must be supported:

1. **Pool Mode (P2 target)**: Tasks specify `judgeMode = POOL`. The `JudgeRegistry` assigns `N` judges (default 3) from the active pool filtered by category expertise.
2. **Quorum Settlement**: Settlement (`judgeAndPay`) requires at least `M` of `N` judges to agree on the same winner and score range (default 2-of-3 majority).

## Design

### Judge Pool Selection

```solidity
struct JudgePoolConfig {
    uint8 poolSize;      // N (e.g., 3)
    uint8 quorum;        // M (e.g., 2)
    uint8 category;      // task category filter
}

function selectJudgePool(
    uint8 category,
    uint256 randomness,
    JudgePoolConfig calldata config
) external view returns (address[] memory judges);
```

- Judges are selected **without replacement** from the active registry.
- Randomness source: `block.prevrandao` for P2; VRF for production.
- Filter: judges must have registered for the task's `category`.

### Quorum Settlement

- Each judge in the pool independently calls `submitJudgement(taskId, winner, score)`.
- Votes are stored on-chain in `judgements[taskId][judge]`.
- Once `quorum` identical `(winner, score +/- tolerance)` votes are reached, the task becomes `settleable`.
- Any party (keeper bot) can call `settleWithQuorum(taskId)` to execute the final `judgeAndPay`-equivalent transfer.

### Incentives & Slashing

- **Reward split**: the 3% judge fee is divided among the `N` judges proportionally to participation.
    - Judges who vote with the majority receive a **majority bonus**.
    - Minority/outlier judges receive a reduced share (or 0 for extreme deviation).
- **Absenteeism**: judges who fail to submit a judgement before `judgeDeadline` are slashed a flat penalty.

## Implementation Plan

### 1. AgentArenaEVM Changes

- Add `JudgeMode` enum: `DESIGNATED`, `SINGLE_POOL`, `MULTI_QUORUM`.
- Add `mapping(uint256 => address[]) public taskJudgePool`.
- Add `submitJudgement(uint256 taskId, address winner, uint8 score)`.
- Add `settleWithQuorum(uint256 taskId)`.

### 2. JudgeRegistry Changes

- Add `selectJudgePool(...)`.
- Add `isEligibleForCategory(address judge, uint8 category)` view.

### 3. Indexer / SDK

- Update `TaskApi` to expose `judgePool: string[]` and `quorumVotes`.
- Update `@gradiences/sdk` to support `judgeMode: 2` in `postTask`.

## Acceptance Criteria

- [ ] `JudgeRegistry` can select a category-filtered pool of `N` active judges.
- [ ] `AgentArenaEVM` stores multi-judge votes and enforces `M-of-N` quorum for settlement.
- [ ] Majority judges receive a higher fee share than outlier/absent judges.
- [ ] Absentee judges in a quorum are slashed.
- [ ] Foundry tests cover 2-of-3 quorum settlement, tie-breaking, and absentee slashing.
