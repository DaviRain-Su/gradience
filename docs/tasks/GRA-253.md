# GRA-253: Evaluator Docker Sandbox MVP

## 目标

替换 `evaluator/runtime.ts` 中返回 `MockSandbox` 的临时实现，为 `code` 和 `api` evaluation types 提供真正可运行的 Docker Sandbox，使 Agent Arena 的 Judge 能客观验证任务结果。

## 现状

- `createSandbox()` 目前直接 `return new MockSandbox(...)`
- `evaluateContent` 和 `evaluateComposite` 为空 TODO
- 这意味着协议能自动结算，但无法客观验证提交内容的正确性
- 架构评审 Finding 5 已标记此问题

## 实现要求

### 1. DockerSandbox 完整实现

- 位于 `apps/agent-daemon/src/evaluator/runtime.ts`（已有 `DockerSandbox` 类占位）
- `code` 类型：
  - 创建临时目录，写入提交代码和测试文件
  - 用 `docker run --rm -v <temp>:/workspace -w /workspace node:20-alpine sh -c "npm install && npm test"`
  - 限时 30s，捕获 stdout/stderr 和 exit code
- `api` 类型：
  - 用 `docker run --rm` 启动一个轻量 HTTP 客户端容器（如 `node:20-alpine`）运行 `fetch` 脚本
  - 验证 endpoint 响应码、响应体结构

### 2. 安全与资源边界

- Docker 命令必须包含 `--network=none`（除非 api 测试需要外网，则使用受限网络）
- CPU 和内存限制（如 `--cpus=1 --memory=512m`）
- 超时后强制 `docker kill`
- 临时目录在完成后清理

### 3. evaluateContent 补全

- 对 `content` 类型：使用 LLM-as-judge 评分（已有 `llm-client.ts` 基础）
- 对 `composite` 类型：顺序执行各子 evaluation，聚合分数

## 验收标准

- [ ] `code` 类型 evaluation 能在 Docker 中真实编译/运行测试
- [ ] `api` 类型 evaluation 能验证真实 endpoint
- [ ] 超时/资源超限场景有明确错误码和日志
- [ ] `runtime.ts` 不再默认返回 MockSandbox（保留 Mock 供测试使用）
- [ ] `npm run build` 在 `apps/agent-daemon` 通过
- [ ] 新增 evaluator 集成测试覆盖 code + api sandbox
