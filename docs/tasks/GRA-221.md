# GRA-221: API Key Management for Agent Access

## Description
OWS uses API keys as the agent access mechanism. The owner creates API
keys scoped to specific wallets and policies. The token (`ows_key_...`)
serves as both authentication AND decryption capability -- the wallet
secret is re-encrypted under an HKDF-derived key from the token.

This means: token stolen without disk = useless. Disk access without
token = can't decrypt. Revoke = delete encrypted copy, token decrypts
nothing.

Our current system uses session tokens from Dynamic SDK. We need to
bridge: user authenticates via Dynamic, then the daemon provisions OWS
API keys for each agent sub-wallet.

## Tasks
1. Use OWS SDK `createApiKey()` to provision agent tokens:
   - Owner auth via Dynamic session -> daemon verifies
   - Daemon calls `createApiKey(name, walletIds, policyIds, passphrase)`
   - Token returned once to frontend, stored securely
2. Create daemon API routes:
   - `POST /api/v1/keys/create` - create API key (requires owner auth)
   - `GET /api/v1/keys/list` - list keys (no tokens shown)
   - `DELETE /api/v1/keys/:id` - revoke key
3. Modify signing routes to accept `ows_key_...` tokens:
   - Detect `ows_key_` prefix -> agent mode (policies enforced)
   - Session token -> owner mode (no policies)
4. Store agent tokens securely:
   - Frontend: in Passkey-protected store (existing)
   - Daemon: in memory only during session
5. Key rotation: create new key, revoke old, update agent config

## Acceptance Criteria
- [ ] Owner can create API keys scoped to specific wallets + policies
- [ ] Agent can sign using `ows_key_...` with policy enforcement
- [ ] Key revocation immediately blocks signing
- [ ] Multiple agents can have independent keys
- [ ] Keys list endpoint never shows raw tokens
